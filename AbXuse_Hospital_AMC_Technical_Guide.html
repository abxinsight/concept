1. Executive Summary

This guide explains how AbXuse computes antimicrobial consumption (AMC) metrics for hospitals. The methodology follows WHO GLASS guidelines to ensure international comparability and alignment with global AMR surveillance standards.

üéØ What AbXuse Does

  1. 1. Accepts consumption data (dispensing, procurement, or prescribing records)
  2. 2. Converts quantities to standardized DDD (Defined Daily Doses)
  3. 3. Normalizes by hospital activity (patient-days, admissions, etc.)
  4. 4. Stratifies by AWaRe category, route, ward, and antimicrobial class
  5. 5. Visualizes trends and benchmarks in interactive dashboards

2. Core Concepts

2.1 What is a DDD?

The Defined Daily Dose (DDD) is the assumed average maintenance dose per day for a drug used for its main indication in adults. DDDs are assigned by the WHO Collaborating Centre for Drug Statistics.

‚ö†Ô∏è Important Clarifications

  • ‚Ä¢ DDD is a statistical unit, not a recommended dose
  • ‚Ä¢ DDD values are route-specific (oral vs parenteral may differ)
  • ‚Ä¢ DDD values are updated annually by WHOCC (version must be tracked)
  • ‚Ä¢ Combination products have special DDD assignment rules
Antimicrobial ATC Code Route DDD Value Unit
Amoxicillin J01CA04 Oral 1.5 g
Amoxicillin J01CA04 Parenteral 3 g
Ceftriaxone J01DD04 Parenteral 2 g
Ciprofloxacin J01MA02 Oral 1 g
Metronidazole J01XD01 Oral 1.5 g

2.2 Consumption Density

Raw DDD counts are meaningless without context. Consumption density normalizes consumption by hospital activity, enabling comparison across hospitals of different sizes and over time.

Metric Formula Use Case
DDD per 100 Patient-Days (DDD √∑ Patient-Days) √ó 100 WHO GLASS recommended; best for inpatient
DDD per 100 Bed-Days (DDD √∑ Bed-Days) √ó 100 When patient-days unavailable
DDD per 100 Admissions (DDD √∑ Admissions) √ó 100 When daily data unavailable; captures intensity per stay
DDD per 1000 Inhabitants per Day (DDD √∑ Population √∑ Days) √ó 1000 Community/national level only

2.3 AWaRe Classification

WHO's AWaRe classification groups antibiotics by their potential for resistance and need for stewardship. The WHO target is ‚â•60% Access antibiotics.

Access

First or second choice for common infections. Narrower spectrum, lower resistance potential.

Examples: Amoxicillin, Metronidazole

Watch

Higher resistance potential. First choice only for specific, limited indications.

Examples: Ciprofloxacin, Ceftriaxone

Reserve

Last resort antibiotics. Should be used only for MDR infections.

Examples: Colistin, Linezolid

‚õî CRITICAL RULE: Numerator-Denominator Match

The coverage of consumption data (numerator) and activity data (denominator) must match exactly. If consumption is hospital-wide, denominators must be hospital-wide. If consumption is ward-level, denominators must be ward-level. Mismatched coverage produces invalid metrics.

4. Calculation Formulas

4.1 DDD Count Calculation

Step 1: Calculate Total Grams

total_grams = quantity_value √ó packsize √ó strength √ó unit_conversion_factor

where unit_conversion_factor converts strength_unit to grams (mg‚Üí0.001, g‚Üí1, MU‚Üívaries)

Step 2: Calculate DDD Count

ddd_count = total_grams √∑ ddd_value_in_grams

where ddd_value_in_grams comes from dim_ddd_value table for the matching substance, route, and ATC/DDD version

üìå Worked Example

Product:Amoxicillin 500mg capsules, box of 100
Quantity dispensed:50 boxes
Route:Oral
DDD (oral):1.5 g

Step 1: total_grams = 50 √ó 100 √ó 500 √ó 0.001 = 2,500 grams

Step 2: ddd_count = 2,500 √∑ 1.5 = 1,666.67 DDDs

‚ö†Ô∏è Special Cases

  • Liquid formulations: Use strength_vol and volume fields.
    total_grams = quantity √ó (strength √∑ strength_vol) √ó volume
  • Combination products: Some have DDD = 1 UD (unit dose). Count units directly.
  • Items vs Packages: If quantity_type = "items", don't multiply by packsize.

4.2 Consumption Density Calculation

DDD per 100 Patient-Days (Recommended)

ddd_per_100_patient_days = (sum(ddd_count) √∑ patient_days) √ó 100

DDD per 100 Admissions

ddd_per_100_admissions = (sum(ddd_count) √∑ admissions) √ó 100

üìå Worked Example

Total DDDs (Q1 2025):15,000
Patient-days (Q1 2025):32,000
Admissions (Q1 2025):4,200

Per 100 Patient-Days: (15,000 √∑ 32,000) √ó 100 = 46.9

Per 100 Admissions: (15,000 √∑ 4,200) √ó 100 = 357.1

4.3 AWaRe Proportions

AWaRe Category Share

access_share = sum(ddd_count WHERE aware = 'Access') √∑ sum(ddd_count) √ó 100
Metric Formula WHO Target
Access Share % Access DDD √∑ Total DDD √ó 100 ‚â• 60%
Watch Share % Watch DDD √∑ Total DDD √ó 100 Monitor
Reserve Share % Reserve DDD √∑ Total DDD √ó 100 Minimize
Oral Share % Oral DDD √∑ Total DDD √ó 100 ‚â• 60%

5. Data Input Workflows

5.1 Product-Level vs Substance-Level Input

Product-Level Entry

User enters specific products with full details (strength, form, pack size). System maps to substance and calculates DDD.

  • ‚úÖ More accurate DDD calculations
  • ‚úÖ Enables formulation analysis
  • ‚ùå Requires product master data
  • ‚ùå More data entry burden

Substance-Level Entry

User enters aggregated totals by substance (e.g., "total grams of amoxicillin"). Simpler but less granular.

  • ‚úÖ Simpler data entry
  • ‚úÖ Works with aggregated reports
  • ‚ùå Less detailed analysis
  • ‚ùå Route must still be specified

5.2 File Upload Processing

Expected File Structure (CSV/Excel)

product_name atc_code route strength strength_unit packsize qty_packages period ward
Amoxicillin 500mg J01CA04 O 500 mg 100 50 2025-01 ICU
Ceftriaxone 1g J01DD04 P 1 g 10 200 2025-01 Surgery

Processing Pipeline

  1. 1. Validation: Check required columns present, data types correct
  2. 2. ATC Lookup: Verify ATC codes exist in reference table
  3. 3. DDD Lookup: Get DDD value for substance + route combination
  4. 4. Calculation: Compute total_grams and ddd_count for each row
  5. 5. AWaRe Mapping: Assign AWaRe category based on ATC code
  6. 6. Aggregation: Sum by desired dimensions (ward, period, etc.)
  7. 7. Storage: Write to fact_consumption table with status = "pending"

7. Validation Rules

Essential Validation (Block if failed)

Rule Check Error Message
ATC Code Valid atc5 exists in dim_substance "Invalid ATC code: {value}"
Route Required route_admin ‚àà ['O','P','Rectal','Inhal',...] "Route of administration required"
DDD Available DDD value exists for substance + route "No DDD defined for {substance} ({route})"
Quantity Positive quantity_value > 0 "Quantity must be positive"
Packsize Present If quantity_type = 'packages', packsize required "Packsize required when entering packages"
Period Match Denominator exists for same hospital + period "No activity data for period {period}"

Plausibility Validation (Warn but allow)

Rule Threshold Warning Message
High Consumption DDD/100 patient-days > 200 "‚ö†Ô∏è Unusually high consumption detected"
ICU Threshold ICU DDD/100 bed-days > 300 "‚ö†Ô∏è ICU consumption may need review"
Reserve High Reserve antibiotics > 10% of total "‚ö†Ô∏è Reserve antibiotic use above typical"
Data Source Change data_source_type differs from previous period "‚ö†Ô∏è Data source changed - trends may not be comparable"

‚õî Data Quality Blocking Rules

  • Missing Denominators: Do not calculate density metrics if no activity data exists for the period. Display "Denominator data required" instead of incorrect values.
  • Coverage Mismatch: If ward-level consumption is entered but only hospital-level denominators exist, block ward-level metrics. Only calculate at matching granularity.
  • Version Tracking: Always store atc_ddd_version_id with consumption records. Different versions may yield different DDD counts for the same data.
